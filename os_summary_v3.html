ما هو نظام التشغيل؟

نظام التشغيل هو برنامج يعمل كوسيط بين المستخدم والأجهزة (Hardware)، يتحكم في تشغيل البرامج وإدارة الموارد.

التعريف الرسمي: مجموعة من برامج النظام التي تتحكم في عمليات الحاسوب معاً.

أمثلة:

UNIX MS-Windows Linux MacOS MS-DOS Android iOS
🎯

أهداف نظام التشغيل

  • تنفيذ برامج المستخدم وتسهيل حل مشاكله
  • جعل استخدام الحاسوب مريحاً وسهلاً
  • استخدام موارد الجهاز بكفاءة عالية
🏗️

مكونات نظام الحاسوب

  1. Hardware: الموارد الأساسية (CPU, RAM, I/O)
  2. OS: يتحكم ويُنسّق استخدام الـ Hardware
  3. التطبيقات: تحدد طريقة استخدام الموارد
  4. المستخدمون: أشخاص أو آلات أو حواسيب أخرى
🔧

Kernel (النواة)

الجزء النشط من نظام التشغيل الذي يعمل طوال الوقت ويتفاعل مباشرة مع الـ Hardware.

  • إدارة العمليات (Process Management)
  • إدارة الذاكرة (Memory Management)
  • نظام الملفات (File System)
  • برامج التشغيل (Device Drivers)
  • إدارة I/O
💬

Shell (مُفسّر الأوامر)

يُسمى command interpreter. مجموعة برامج للتفاعل مع التطبيقات.

  1. المستخدم يكتب الأوامر
  2. Shell يُفسّر الأوامر
  3. يُرسل تعليمات لنظام التشغيل
  4. النظام يُنفّذ عبر التطبيقات
  5. النتيجة تُعاد للمستخدم
📋

  • 💾 Memory Management — إدارة الذاكرة

1940s–1950s

1. لا يوجد نظام تشغيل

البرامج تُنفَّذ يدوياً بـ punch cards. برنامج واحد في وقت واحد. المشغّل الإنسان هو نظام التشغيل!

يدوي بالكاملبرنامج واحد فقط
1950s–1960s

2. Mainframe Systems

أول نظام تشغيل حقيقي (Resident Monitor). يقلّل وقت الإعداد بتجميع المهام المتشابهة. ينقل التحكم تلقائياً من مهمة لأخرى.

✅ مميزات
  • قدرة معالجة ضخمة
  • قابل للتوسع
  • موثوقية عالية
  • أمان قوي
❌ عيوب
  • تكلفة عالية جداً
  • معقد التشغيل
  • غير مرن
1960s–1970s

3. Batch Processing OS

يقبل أكثر من مهمة ويجمّعها حسب المتطلبات المتشابهة. برنامج واحد في وقت واحد. جدولة المهام حسب الأولوية والموارد.

✅ مميزات
  • كفاءة موارد عالية
  • إنتاجية مرتفعة
  • تقليل الأخطاء البشرية
  • فعّال من حيث التكلفة
❌ عيوب
  • لا يحل مشاكل معقدة
  • مشاكل أمنية
  • يمكن مقاطعته
  • بطيء وغير فعال
1960s–1970s

4. Multiprogramming OS

تُحمَّل عدة برامج في الذاكرة. عندما يحتاج برنامج لـ I/O، يُنفّذ برنامج آخر. الـ CPU لا يتوقف أبداً ما دامت هناك مهام.

المفهوم الأساسي: Job Pool في الـ Disk ← OS يحمّل مجموعة منها في RAM ← عند انتظار برنامج، يُنفّذ برنامج آخر
✅ مميزات
  • استغلال أمثل للـ CPU
  • يدعم مستخدمين متعددين
  • يزيد الإنتاجية (Throughput)
❌ عيوب
  • قد يسبب Thrashing
  • يحتاج إدارة ذاكرة
  • يتطلب خوارزميات جدولة
1960s–1970s

5. Real-Time OS (RTOS)

لتطبيقات ذات مواعيد نهائية محددة (deadlines). يُستخدم في: أنظمة تحكم السيارات، الطائرات، الروبوتات، الفضاء.

🔴 Hard Real-Time

ضمان إكمال المهام في وقتها تماماً. أي تأخير = فشل. مثل: أنظمة الطيران.

🟡 Soft Real-Time

أقل صرامة. المهام الحرجة تأخذ أولوية. يمكن دمجه مع أنظمة أخرى.

1970s–1980s

6. Time-Sharing / Multitasking OS

امتداد للـ Multiprogramming. التبديل السريع بين المهام. مستخدمون متعددون في نفس الوقت. يستخدم مفهوم الـ Virtual Memory. أمثلة: UNIX, CTSS, MULTICS.

✅ مميزات
  • كل مهمة تأخذ فرصة متساوية
  • تقليل توقف الـ CPU
❌ عيوب
  • مشاكل موثوقية
  • أمان البيانات
  • مشاكل التواصل
1980s–1990s

7. Multiprocessor OS (Parallel OS)

أكثر من معالج يشتركون في Bus وClock وأحياناً الذاكرة. ينفّذون مهام متعددة في نفس الوقت.

Symmetric (SMP)

كل المعالجات متساوية. تشاركية في الـ Memory. أي معالج يمكنه تنفيذ أي مهمة.

أكثر مرونةأغلى تصميماً

Asymmetric (AMP)

Master-Slave. المعالج الرئيسي يُجدوِل للآخرين. الـ Master يتحكم في كل شيء.

أرخصإذا فشل Master فشل الكل
2000s–Present

8. Distributed OS

أجهزة مختلفة مرتبطة بشبكة. كل جهاز له معالجه وذاكرته المحلية. نظم التشغيل تتعاون لإدارة موارد الشبكة.

Client-Server

Peer-to-Peer (P2P)

⚡ ملخص سريع — أنواع أنظمة التشغيل
🌟 أهم نقطة أنظمة التشغيل تطورت من Manual (بدون OS) → Mainframe → Batch → Multiprogramming → Real-Time → Time-Sharing → Multiprocessor → Distributed.
❓ أشهر سؤال ما الفرق بين Batch وMultiprogramming؟ — Batch: برنامج واحد في وقت واحد. Multiprogramming: عدة برامج في الذاكرة، عند انتظار أحدها يُنفَّذ غيره.
⚡ Hard vs Soft Hard Real-Time: أي تأخير = فشل (طيران/فضاء). Soft Real-Time: أولوية للمهام الحرجة لكن التأخير مقبول.

ما هي العملية (Process)؟

برنامج يُنفَّذ حالياً على الحاسوب. نسخة نشطة تستخدم موارد النظام (CPU, Memory, I/O).

أقسام العملية في الذاكرة:
  1. Text/Code Segment: الكود التنفيذي
  2. Data Segment: المتغيرات العامة والثابتة
  3. Heap: التخصيص الديناميكي للذاكرة
  4. Stack: نداءات الدوال والمتغيرات المحلية
  5. Memory-mapped Region: المكتبات المشتركة

📋

PCB (Process Control Block)

هيكل بيانات يحتفظ بمعلومات كل عملية:

  • حالة العملية (Process State)
  • أولوية العملية (Priority)
  • الموارد المستخدمة
  • رقم تعريف العملية (PID)
🚀

الـ Dispatcher

المكوّن المسؤول عن نقل التحكم من الـ Scheduler إلى العملية المختارة فعلياً.

  • Context Switching: حفظ وتحميل حالة العملية
  • Resource Allocation: تخصيص الموارد
  • Control Transfer: نقل التحكم للـ CPU
Dispatch Latency: الوقت اللازم لإيقاف عملية وتشغيل أخرى
⚡

الـ Interrupt (المقاطعة)

إشارة تُرسَل للـ CPU تقول: "توقف عما تفعله، هناك شيء مهم يحتاج انتباهك فوراً".

🔧 Hardware Interrupt

تأتي من أجهزة خارجية. مثل: ضغطة كيبورد، انتهاء طباعة، إشارة شبكة.

💻 Software Interrupt

تأتي من البرامج. مثل: System Calls، أخطاء تقسيم على صفر.

⚡ ملخص سريع — إدارة العمليات
🌟 أهم نقطة العملية تمر بـ 5 حالات: New → Ready → Running → Waiting → Terminated. الـ PCB هو "هوية" كل عملية في النظام.
❓ أشهر سؤال ما الفرق بين Process وProgram؟ — البرنامج كود ساكن على الديسك. العملية برنامج يُنفَّذ حالياً ويستخدم موارد (CPU, RAM, I/O).
⚙️ فرق مهم Scheduler يختار العملية. Dispatcher ينقل التحكم فعلياً للـ CPU. Dispatch Latency = وقت هذه العملية.

Preemptive vs Non-Preemptive

Preemptive: الـ OS يمكنه إيقاف عملية جارية وتشغيل أخرى.
Non-Preemptive: لا يمكن إيقاف العملية حتى تنتهي أو تتخلى عن الـ CPU.
📐

الصيغ الأساسية

Turnaround = Completion - Arrival
Waiting = Turnaround - Burst Time
Response = First Run - Arrival
Avg Waiting = Σ Waiting / n

2️⃣ SJF — Shortest Job First

Non-Preemptive / Preemptive

ينفّذ العملية ذات أقل Burst Time أولاً. النسخة الـ Preemptive تُسمّى SRTF (Shortest Remaining Time First).

✅ يعطي أقل Average Waiting Time نظرياً (Optimal).
⚠️ مشكلة: لا يمكن معرفة Burst Time مسبقاً في الواقع (يحتاج تنبؤ).

3️⃣ Priority Scheduling

Preemptive / Non-Preemptive

لكل عملية أولوية. العملية ذات الأولوية الأعلى تُنفَّذ أولاً. (رقم أصغر = أولوية أعلى عادةً)

🚨 Starvation: العمليات ذات الأولوية المنخفضة قد لا تُنفَّذ أبداً إذا وصلت عمليات عالية الأولوية باستمرار.

🧵 الخيوط (Threads)

ما هو الخيط؟

تدفق تنفيذ واحد ويُعتبر وحدة أساسية من وحدات استخدام الـ CPU. الخيط هو تسلسل من التعليمات داخل عملية.

المشترك بين الخيوط:
  • Code Segment
  • Data Segment

الخاص بكل خيط:
  • Thread ID
  • Program Counter (PC)
  • Registers
  • Stack

نماذج الخيوط (Threading Models)

User-Level Threads

في مساحة المستخدم. الـ Kernel لا يعرف بها. سريعة الإنشاء. إذا انتظر خيط → تنتظر العملية كلها.

Kernel-Level Threads

يديرها الـ Kernel مباشرة. أبطأ في الإنشاء. إذا انتظر خيط، يُنفَّذ خيط آخر من نفس العملية.

Hybrid Threads

يجمع مميزات النوعين. Kernel Threads تحمل User Threads متعددة.

🔒 الـ Deadlock
الشروط الأربعة + طرق الكشف والمنع والتجنب
🔒

ما هو الـ Deadlock؟

حالة تكون فيها مجموعة من العمليات في حالة انتظار دائم — كل عملية تنتظر موارد تحتلها عملية أخرى تنتظر هي الأخرى.

مثال: P1 تمتلك R1 وتنتظر R2. P2 تمتلك R2 وتنتظر R1. → توقف تام!
⚠️

الشروط الأربعة للـ Deadlock (Coffman Conditions)

يجب أن تتحقق جميعها في نفس الوقت لحدوث Deadlock:

1️⃣ Mutual Exclusion

المورد لا يمكن استخدامه من أكثر من عملية في نفس الوقت.

2️⃣ Hold and Wait

عملية تمتلك موارد وتنتظر موارد إضافية عند عمليات أخرى.

3️⃣ No Preemption

لا يمكن أخذ المورد من العملية بالقوة. يُحرَّر طوعاً فقط.

4️⃣ Circular Wait

سلسلة دائرية: P1 تنتظر P2 تنتظر P3 تنتظر P1.

🛡️

طرق التعامل مع الـ Deadlock

🔍 Detection (الكشف)

Resource Scheduler يتتبع الموارد. بعد الكشف: إنهاء العمليات أو سحب موارد منها.

🚫 Prevention (المنع)

نمنع أحد الشروط الأربعة من التحقق قبل تنفيذ أي معاملة.

🏃 Avoidance (التجنب)

نستخدم Wait-for Graph لتجنب الـ Deadlock. مفيد للأنظمة الصغيرة.

😵

Starvation

عملية ذات أولوية منخفضة تنتظر للأبد لأن دائماً تأتي عمليات ذات أولوية أعلى.

الحل: Aging — نزيد أولوية العملية كلما طال انتظارها.
🔄

IPC (Inter-Process Communication)

العمليات تتواصل مع بعضها عبر IPC.

  • Shared Memory — مساحة ذاكرة مشتركة
  • Message Passing — إرسال رسائل

Banker's Algorithm (خوارزمية المصرفي)

خوارزمية Deadlock Avoidance — تمنع الـ Deadlock قبل حدوثه بتحديد ما إذا كانت الحالة "آمنة" (Safe State) قبل منح الموارد.

المفاهيم الأساسية:
  • Max: أقصى ما تحتاجه العملية من كل نوع موارد
  • Allocation: ما تمتلكه حالياً
  • Need = Max − Allocation
  • Available: الموارد المتاحة في النظام
Safe State = Safe Sequence موجود
إذا كان هناك ترتيب للعمليات بحيث كل عملية يمكن إنهاؤها باستخدام الموارد المتاحة + موارد العمليات التي أنهت قبلها → الحالة آمنة.

📝 مثال محلول — Banker's Algorithm

3 أنواع موارد: A=10, B=5, C=7  |  5 عمليات

Process Alloc (A,B,C) Max (A,B,C) Need (A,B,C)
P00,1,07,5,37,4,3
P12,0,03,2,21,2,2
P23,0,29,0,26,0,0
P32,1,12,2,20,1,1
P40,0,24,3,34,3,1
Available = (10−7, 5−2, 7−5) = (3, 3, 2)
  1. Available=(3,3,2). P1 Need=(1,2,2) ≤ Available ✅ → أنجز P1. Available=(3,3,2)+(2,0,0)=(5,3,2)
  2. P3 Need=(0,1,1) ≤ (5,3,2) ✅ → أنجز P3. Available=(5,3,2)+(2,1,1)=(7,4,3)
  3. P4 Need=(4,3,1) ≤ (7,4,3) ✅ → أنجز P4. Available=(7,4,3)+(0,0,2)=(7,4,5)
  4. P0 Need=(7,4,3) ≤ (7,4,5) ✅ → أنجز P0. Available=(7,4,5)+(0,1,0)=(7,5,5)
  5. P2 Need=(6,0,0) ≤ (7,5,5) ✅ → أنجز P2.
Safe Sequence: P1 → P3 → P4 → P0 → P2 ✅ الحالة آمنة!
⚡ ملخص سريع — الـ Deadlock
🌟 أهم نقطة الـ Deadlock يحتاج 4 شروط في آن واحد: Mutual Exclusion + Hold & Wait + No Preemption + Circular Wait. كسر أي شرط = لا Deadlock.
❓ أشهر سؤال ما الفرق بين Prevention وAvoidance؟ — Prevention: نمنع أحد الشروط الأربعة مسبقاً. Avoidance: نسمح بالتشغيل ولكن نتحقق من Safe State قبل كل تخصيص (Banker's).
🏦 Banker's Need = Max − Allocation. إذا Need ≤ Available → آمن. Safe Sequence = الدليل على السلامة.
💾 إدارة الذاكرة
Swapping، Address Binding، الـ MMU، وتقنيات التخصيص
🧠

لماذا نحتاج إدارة الذاكرة؟

الـ RAM تتيح للـ CPU الوصول المباشر. هي Volatile (تُمحى عند إيقاف التشغيل). يجب تحميل الـ OS والبرامج فيها عند كل تشغيل.

الـ CPU يصل مباشرة لـ RAM فقط — كل شيء آخر (Disk) يجب نقله للـ RAM أولاً.
🔗

Address Binding (ربط العناوين)

ربط عناوين البرنامج المنطقية بعناوين فيزيائية حقيقية في الذاكرة.

1. Compile-Time

العنوان يُحدَّد أثناء الترجمة. إذا تغير المكان → يجب إعادة الترجمة.

2. Load-Time

العنوان يُحدَّد عند التحميل في الذاكرة. Relocatable Code. يمكن تحميله في أي مكان.

3. Execution-Time

يحدث أثناء التشغيل. يحتاج دعم Hardware (MMU). الأكثر شيوعاً.

🗺️

Logical vs Physical Address

Logical (Virtual) Address: ينشئها الـ CPU أثناء التشغيل. لا وجود فيزيائي لها.
Physical Address: العنوان الحقيقي في الـ RAM. المستخدم لا يتعامل معها مباشرة.

الـ MMU (Memory Management Unit) يحوّل Logical → Physical

🔄

Swapping

نقل عملية من الـ RAM إلى Backing Store (Disk) مؤقتاً ثم إعادتها.

  • Roll out: إخراج عملية ذات أولوية منخفضة
  • Roll in: إدخال عملية ذات أولوية أعلى
  • وقت النقل يتناسب مع حجم الذاكرة المنقولة
🏗️

تقنيات تخصيص الذاكرة

النوعالوصفمميزاتعيوب
Contiguous تخصيص كتلة متصلة واحدة بسيط وسريع الوصول Fragmentation
Non-Contiguous كتل غير متصلة لا External Fragmentation يحتاج Page Table

Fragmentation (التجزئة)

Internal Fragmentation

الذاكرة المخصصة أكبر من المطلوب. المساحة الزائدة داخل الـ Partition لا تُستخدم.

مثال: Partition=4MB, Process=1MB → داخلي = 3MB

External Fragmentation

مساحات حرة موجودة لكنها غير متصلة، لا يمكن استخدامها لعملية كبيرة.

P5=7MB لكن 7MB متفرقة → لا يمكن تخصيصها!
📐

خوارزميات التخصيص: First-Fit / Best-Fit / Worst-Fit

الخوارزميةالوصفالخاصية
First Fit أول فراغ كافٍ من البداية أسرع — يتوقف عند أول match
Best Fit أصغر فراغ يتسع للعملية أقل هدراً — يبحث في كل القائمة
Worst Fit أكبر فراغ متاح يترك أكبر بقية — يبحث في كل القائمة

📝 مثال على الفرق:

الفراغات المتاحة: 100K, 500K, 200K, 300K, 600K — عملية 212K

First Fit:
أول فراغ ≥ 212K → 500K
يبقى 288K
Best Fit:
أصغر ≥ 212K → 300K
يبقى 88K فقط
Worst Fit:
أكبر → 600K
يبقى 388K كبقية

خوارزميات إحلال الصفحات (Page Replacement)

عندما تكون الذاكرة ممتلئة ونحتاج تحميل صفحة جديدة، نُحدد أي صفحة نُخرج (Victim Page). الهدف: تقليل عدد الـ Page Faults.

1️⃣ FIFO

أول صفحة دخلت → أول صفحة تخرج. بسيط لكن ليس الأمثل. قد يعاني من Belady's Anomaly (زيادة الـ Frames تزيد Page Faults!).

2️⃣ LRU

Least Recently Used — أقل صفحة استُخدمت مؤخراً تخرج. يعتمد على سجل الاستخدام. لا يعاني من Belady's Anomaly. عملي وشائع.

3️⃣ Optimal

تخرج الصفحة التي لن تُستخدم لأطول وقت في المستقبل. أقل عدد Page Faults ممكن. نظري فقط — غير قابل للتطبيق.

📝 مثال مقارن — 3 Frames، Reference String: 7,0,1,2,0,3,0,4,2,3,0,3,2

الخوارزمية عدد Page Faults الصفحة المُستبدَلة عند 2,3,4 (مثال)
FIFO 9 ❌ (الأسوأ) أول صفحة دخلت (بغض النظر عن الاستخدام)
LRU 8 ⚠️ (جيد) الصفحة التي لم تُستخدم لأطول وقت
Optimal 5 ✅ (الأمثل) الصفحة التي لن تُستخدم لأطول وقت مستقبلاً
⚠️ Belady's Anomaly: في FIFO فقط — زيادة عدد الـ Frames أحياناً تزيد Page Faults! LRU وOptimal لا يعانيان من هذه المشكلة.
Page Fault Rate ↓ = أداء أفضل  |  Optimal = المرجع للمقارنة  |  LRU = الأفضل عملياً
⚡ ملخص سريع — إدارة الذاكرة
🌟 أهم نقطة الـ CPU يصل مباشرة للـ RAM فقط. الـ MMU يحوّل Logical Address → Physical Address. الـ Swapping ينقل عمليات بين RAM وDisk.
❓ أشهر سؤال ما الفرق بين Internal وExternal Fragmentation؟ — Internal: مساحة ضائعة داخل الـ Partition المخصص. External: مساحات حرة موجودة لكنها مفككة ولا يمكن استخدامها.
🔁 Page Replacement Optimal = الأقل فaults لكن نظري. LRU = الأفضل عملياً. FIFO = الأبسط لكن يعاني Belady's Anomaly.

Paging (التصفيح)

تقسيم الذاكرة المادية إلى Frames متساوية، والذاكرة المنطقية إلى Pages بنفس الحجم.

Page Size = Frame Size = 2ⁿ
مثال: n=2 → Size=4 → عناوين 00, 01, 10, 11
✅ لا External Fragmentation
⚠️ قد يوجد Internal Fragmentation
🗺️

ترجمة العناوين في Paging

Logical Address

= Page Number + Page Offset
  • Page Number: رقم الصفحة
  • Page Offset: الموضع داخل الصفحة

Physical Address

= Frame Number + Page Offset
  • Frame Number: من الـ Page Table
  • Page Offset: نفس الـ Logical Offset
الـ PTBR (Page Table Base Register): يحتوي على عنوان بداية الـ Page Table في الذاكرة.
📐

📝 مثال محلول — ترجمة عنوان Paging

البيانات: Process P=195 bytes، Page Size=64 bytes
Page Table: Page 0→Frame 1, Page 1→Frame 3, Page 2→Frame 4, Page 3→Frame 2
  1. حساب عدد الصفحات: 195/64 = 3.05 → يحتاج 4 pages
  2. Logical Address مثلاً: 100
    Page Number = 100 ÷ 64 = 1 (الصفحة 1)
    Page Offset = 100 mod 64 = 36
  3. من الـ Page Table: Page 1 → Frame 3
  4. Physical Address = Frame 3 × 64 + 36 = 192 + 36 = 228
Physical = (Frame Number × Page Size) + Page Offset
= (3 × 64) + 36 = 228
🗂️

Segmentation (التقطيع)

تقسيم العملية إلى Segments (كتل غير متساوية) تمثّل الوحدات المنطقية: code, data, stack.

Segment Table يحتوي لكل Segment:
  • Base: عنوان البداية في الذاكرة
  • Limit: طول الـ Segment
Logical Address = (Segment#, Offset)
Physical = Base[Segment#] + Offset
يجب أن Offset < Limit
🔀

Paging + Segmentation (Hybrid)

يجمع الأفضلين: كل Segment يحتوي على Pages. مستويين من الـ Mapping.

  • Segments تمثّل الوحدات المنطقية (code, data, stack)
  • Pages تُستخدم من جهة الـ OS لسهولة إدارة الذاكرة
  • لكل Segment: Base + Bound + Page Table خاصة
✅ يتجنب External Fragmentation (من Paging) + يُعطي رؤية منطقية للمستخدم (من Segmentation)
⚡ ملخص سريع — Paging & Segmentation
🌟 أهم نقطة Paging يقضي على External Fragmentation لكن يبقى Internal. المعادلة الأساسية: Physical = (Frame# × Page Size) + Offset.
❓ أشهر سؤال ما الفرق بين Paging وSegmentation؟ — Paging: أحجام متساوية (Pages/Frames)، يديره OS. Segmentation: أحجام متغيرة تعكس بنية البرنامج (code, data, stack)، يديره المبرمج.
🔢 المعادلة Page Number = LA ÷ Page Size  |  Offset = LA mod Page Size  |  PA = Frame# × Page Size + Offset
